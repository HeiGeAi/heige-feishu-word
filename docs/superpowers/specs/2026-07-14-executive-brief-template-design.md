# executive-brief 双页模板设计

日期：2026 年 7 月 14 日

状态：待用户书面审核

## 1．背景与问题

`heige-feishu-word` 当前标准样本结构完整，但视觉与沟通效率明显弱于「酱油提前转正与岗位调整｜双页沟通版」。两份飞书文档的实时回读结果显示：

| 指标 | 酱油版 | 当前 Demo |
|---|---:|---:|
| XML 内容字符数 | 约 5600 | 约 8500 |
| Callout | 8 | 2 |
| 文字着色 | 22 | 15 |
| 加粗强调 | 25 | 17 |
| 二级标题 | 3 | 9 |
| PDF 页数 | 2 | 3 |

问题不是单纯缺少颜色，而是当前 Demo 使用了「长报告」表达方式：章节多、解释多、组件之间缺少强弱关系。酱油版使用的是「管理沟通稿」表达方式：先给结论，再用对照、边界、数字、行动和可直接使用的话术完成决策闭环。

## 2．目标

本轮新增一个可复用的 `executive-brief` 模板，并用它重做标准 Demo。

目标包括：

1. 复用酱油版的视觉语言，而不是复制酱油版的具体业务内容。
2. 将 Demo 压缩为双页管理简报，读者可在 30 秒内抓住结论、矛盾、边界、指标和行动。
3. 保留飞书原生结构、原生画板和可编辑能力。
4. 将颜色和装饰沉淀为模板规则，不在某一份 Demo 中写死颜色值。
5. 保留现有 `editorial-forest`，让调用方可以明确选择不同交付风格。

## 3．非目标

本轮不实现以下能力：

1. 素材解析与附件上传。
2. Claude Code、Codex、OpenCode、Hermes 的完整适配层。
3. 任意文档自动保证双页。
4. 通用拖拽式排版器。
5. GitHub 正式开源发布。

双页约束只对本次标准样本生效。通用模板负责提高成功概率，最终页数仍由 PDF 质量门检查。

## 4．视觉系统

### 4.1 色彩语义

模板使用酱油版已经验证过的五色体系：

| 语义 | 主色 | 浅色背景 | 用途 |
|---|---|---|---|
| 主结论 | 深海军蓝 | 浅蓝 | 标题、核心结论、主路径 |
| 已验证 | 绿色 | 浅绿 | 正向能力、通过项、完成状态 |
| 待评审 | 橙色 | 浅橙 | 条件、注意事项、阶段任务 |
| 禁止或风险 | 红色 | 浅红 | 边界、失败条件、安全红线 |
| 辅助信息 | 灰色 | 浅灰 | 说明、来源、次级文字 |

颜色只表达语义，不作为随机装饰。同一种颜色在整篇文档中必须保持同一种含义。

### 4.2 组件密度

标准样本目标值：

1. Callout 不少于 6 个，不超过 9 个。
2. 文字着色不少于 20 处。
3. 加粗强调不少于 20 处。
4. 二级标题不超过 5 个。
5. 只保留 1 个主画板。
6. 不允许整段正文全部使用同一种强调色。

### 4.3 组件样式

1. 文档开头使用深色结论区，呈现「结论、边界、试点周期」。
2. 成对信息使用双栏彩色卡片，例如「为什么值得做」与「当前不解决什么」。
3. 数字指标使用大数字、彩色标签和一句解释，不堆成长段。
4. 表格首列使用语义底色，形成「可以做、需要评审、不能承诺」式边界感。
5. 行动顺序使用居中的步骤标签，避免普通待办列表占据过多空间。
6. 章节标题使用短标题和英文 Eyebrow，减少连续九个同权重标题。

## 5．双页内容结构

### 第 1 页｜为什么做，以及产品边界

1. 标题与一句话定位。
2. 深色管理结论区：建议进入 14 天 MVP 试点，但不做万能文档平台。
3. 双栏对照：当前业务痛点与 MVP 回应。
4. 四个设计边界：统一 Body、四类 Agent、三个试点场景、十四天验证。
5. 三层能力边界表：可以直接交付、需要人工评审、本阶段不承诺。
6. 一句话定性：产品解决的是企业文档交付最后一公里，不替代业务判断。

### 第 2 页｜怎么交付，以及如何判断继续

1. 紧凑型原生画板：材料进入、内容整理、视觉编译、飞书发布、质量验收、完成交付。
2. 三个试点场景：管理简报、会议到行动、项目复盘。
3. 四个验收指标：素材完整率、首版生成时间、人工调整时间、Agent 兼容性。
4. 双栏沟通卡：TO BUSINESS 与 TO TECH。
5. 推进顺序：审核样本、冻结契约、准备材料、启动试点。

## 6．模板与数据契约

### 6.1 模板选择

`theme` 保持字符串形式，新增值：

```json
{
  "theme": "executive-brief"
}
```

渲染器根据模板选择颜色、标题、卡片、指标和表格表现。内容 Body 不直接保存 RGB 色值。

### 6.2 语义色字段

在不破坏旧 Body 的前提下，新增可选语义字段：

```json
{
  "tone": "primary | success | warning | risk | neutral"
}
```

支持范围：

1. `callout` section。
2. `grid.items[*]`。
3. `metrics.items[*]`。
4. `table.row_tones[*]`，与 `rows` 一一对应。

未知 tone、数量不匹配或未被渲染的字段必须明确失败，不能静默忽略。

### 6.3 模板边界

`editorial-forest` 保持现有行为。`executive-brief` 只改变视觉编译规则和标准样本内容，不改变飞书发布、画板创建、PDF 导出和交付回执接口。

## 7．渲染架构

新增独立主题配置层：

```text
Document Body
    ↓ validate
Theme Registry
    ↓ resolve semantic tone
XML Renderer + SVG Renderer
    ↓
Feishu Docx XML + editable whiteboard
```

主要边界：

1. `model.py` 负责模板名、tone 和行数一致性校验。
2. `themes.py` 只负责语义色到飞书颜色属性的映射。
3. `xml_renderer.py` 负责选择组件表现，不保存业务特例。
4. `svg_renderer.py` 只增加紧凑型布局，不改变节点可编辑要求。
5. 标准样本 Body 负责业务内容与 tone，不保存具体 RGB。

## 8．错误处理

以下情况必须返回机器可读错误：

1. 未知模板。
2. 未知 tone。
3. `row_tones` 与表格行数不一致。
4. 紧凑画板文字溢出。
5. 双页 PDF 出现空白页、孤立标题、跨页表格或超过两页。

本地编译失败时不得覆盖上一次成功产物。飞书覆盖失败时保留本地 XML、PDF 和准确错误边界。

## 9．测试与验收

### 自动测试

1. 旧 `editorial-forest` 快照保持通过。
2. `executive-brief` 颜色语义和组件结构测试通过。
3. tone、表格行数和未知字段的失败测试通过。
4. 画板几何检查为 0 错误、0 警告。
5. 飞书回读关键锚点 100％ 命中。
6. 飞书画板业务标签 100％ 命中，图片节点为 0。

### 人工视觉验收

1. 5 秒内能看到管理结论和产品边界。
2. 30 秒内能讲清「为什么做、怎么做、怎么判断继续」。
3. 颜色有明确语义，没有彩虹化和整段染色。
4. 视觉语言与酱油双页版属于同一套设计系统。
5. PDF 恰好 2 页 A4，无空白页、孤立标题、表格断裂和文字裁切。
